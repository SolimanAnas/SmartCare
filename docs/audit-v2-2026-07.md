## Part 2 — Roadmap critique

**Items that should be removed or reframed:**

- **CQ §5 (version string)** — remove. The content-hash precache version solved the real problem; a marketing version string is a 10-minute nicety, not roadmap material.
- **AUTH §2 (admin session expiry)** — demote to polish. `sendToLogin()` on 401/403 killed the actual failure mode.
- **MED §4 (printable cards)** — postpone indefinitely. Print CSS already exists where users actually print (exam review). No evidence of demand for card printing; the effort is better spent on content versioning.
- **AI section** — keep parked per owner decision; do not let it absorb effort before Medical Workflow.
- **CQ §2 (partials)** — reframe. The roadmap targeted head/footer duplication across 37 pages, but the audit shows the *real* duplication is the exam-review family: ~10 pages each carrying ~1,800 lines of a nearly identical inline framework (own theme system, own `escapeHtml`, own print CSS). Extracting *that* into a shared `exam-review.js/css` is a far bigger win than templating footers.
- **Offline §5** — keep as a checklist, not active work; it is externally blocked and the honest status block already says exactly what the next owner-actions are.

**Items whose priority should rise:**

- **SEO §2 (canonical domain)** — was Low; now **Critical**. It transitively blocks `assetlinks.json`, the sitemap, and correct OG metadata. Everything store-related funnels through this one decision.
- **DB §2 (sync merge)** — was Medium; should be **Critical-adjacent**. The app now *feels* reliable (offline verified, error states everywhere), which makes silent whole-blob data loss the most damaging remaining failure mode for a study app.
- **MED §1 (content versioning)** — unchanged priority on paper but now the weakest pillar *relative to everything else*. The engineering scaffolding (meta.js per chapter) makes adding `reviewedBy/reviewedAt/sourceGuideline` fields cheap; the hard part is the human review process, which is exactly why it should start early.

## Part 4 — Newly discovered opportunities (not in the original roadmap)

1. **Repo slimming with history rewrite** — `pdf_sections/` (103 MB of copyrighted source PDFs) makes `.git` 122 MB. Move to private storage, delete, and BFG/filter-repo the history. Every clone and CI checkout pays this today.
2. **Exam-review framework extraction** — the 10-page family duplicating ~1,800 lines each is the largest single deduplication opportunity in the repo (bigger than the roadmap's partials item).
3. **Offline smoke test in CI** — the environment already has a pinned Chromium for Playwright; a CI job that serves the tree, registers the SW, and asserts zero pre-cache failures would make the offline guarantee regression-proof instead of session-verified.
4. **`escapeHtml` consolidation** — 12+ copies with *different* escaping completeness (some skip quotes) — this is a correctness issue, not just hygiene. One `src/dom-utils.js`.
5. **Drug-calculator dead-code fold** — `drug-calculator.html` overrides `boot()`/`pickDrug()` from `drug-calculator.js`; the .js file's modal-picker half targets elements that exist only as hidden dummy divs. Merge into one file, delete the rest.
6. **Runtime cache pruning** — SWR caches every visited page/image into the versioned cache with no cap. Add an LRU cap or a separate bounded runtime cache.
7. **Local error ring buffer** — reframe TD §2 without a third-party service: `window.onerror` → capped localStorage buffer → viewer in the admin console. Privacy-consistent, offline-friendly.
8. **Supabase migrations as code** — commit `supabase/migrations/*.sql` so DB §1 stops being dashboard folklore.
9. **Screenshot pipeline** — the `tab*.png` shots are now *stale* (tablet layout changed this cycle) as well as heavy; regenerate at ~1280 px and compress (24 MB → ~6 MB), killing two items at once.
10. **Stale-screenshot honesty** — until regenerated, the store screenshots advertise a layout that no longer exists; worth fixing before any store submission.

## Part 7 — Final verdict

**Is SmartCare better than at the previous audit?** Substantially. The offline story
went from *silently broken* (chapter pages never registered the SW at all — found and
fixed this cycle) to *proven complete*. The Flask attack surface is gone. The repo lost
~60 MB of shipped media. Five of six workstreams that touch users daily are done.

**Biggest improvements:** offline/PWA pipeline (generated, drift-checked, verified);
admin console rebuilt on a real security boundary; the two latent-bug discoveries
(SW registration, icon paths) that only surfaced because the roadmap work forced
end-to-end verification.

**Biggest weaknesses now:** (1) medical content governance — no version/review
metadata on clinical content in an app whose whole premise is clinical accuracy;
(2) whole-blob sync — the one remaining way a diligent user can silently lose data;
(3) accessibility of the clinical tools — zero ARIA on the pages a medic uses one-handed
in the field; (4) testing depth — the frontend has no automated regression net in CI.

**One month, forced to choose:** Week 1 — domain decision, migration-as-code, SRI/CSP.
Week 2 — sync merge hardening + Playwright/offline CI. Week 3 — tool-page ARIA +
reduced-motion + screenshot regeneration. Week 4 — content-versioning schema + port
the Prometric SRS to chapter flashcards. That sequence converts the app from
"technically excellent PWA" to "defensible medical study product."

**Google Play ready?** Not yet — but close. In-repo prerequisites are met except the
canonical domain and fresh screenshots; the remaining blockers (signing cert,
`assetlinks.json`, Play Console) are account-level, ~2 focused weeks after the domain
decision. An accessibility pass before submission is strongly advised.

**Apple App Store ready?** Further out. Requires the PWABuilder wrapper, an Apple
developer account, physical-device verification of the new iOS meta/splash work
(never tested on hardware), and review-proofing. The flashcard/quiz engines help the
"more than a website" argument; the medical-content disclaimer story must be airtight.

**Remaining blockers, in order:** canonical domain → Play signing + assetlinks →
stale screenshots → accessibility pass → (Apple only) wrapper + device testing.
